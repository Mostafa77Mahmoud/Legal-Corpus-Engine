## لكن لسه عندي ملاحظتين

### 1. Multi-ref design ضعيف

هو حالياً عامل:

```json
{
  "article_numbers": [19,22],
  "target_article_id": null
}
```

أنا مش بحب التصميم ده.

الأفضل:

```json
{
  "article_numbers": [19,22],
  "target_article_ids": [
      "EG_ESIGN_019",
      "EG_ESIGN_022"
  ]
}
```

ليه؟

عشان كده كده Stage 6 عنده الـ lookup table.

فحلها وقت الـ build أرخص من حلها وقت الـ query.

اسأله لو حابب:

> Can target_article_ids be generated for multi-reference citations as well?

---

### 2. confidence_score

دي أهم نقطة متبقية.

هو قال:

```json
"confidence_score": null
```

وده معناه إن جزء من الـ auditability لسه ناقص.

في V7 إحنا كنا عايزين كل مادة يبقى معروف:

```json
{
   "confidence_score": 0.94,
   "extraction_source": "pymupdf"
}
```

مش شرط الآن توقف المشروع بسببها.

لكن قبل ما تدخل القانون المدني أو التجاري لازم تتأكد إن:

```bash
--stages 1 2 3 4 5 6 7
```

بيملأ الحقول دي فعلاً.

---

# رأيي الحالي

لو كنت مكانك:

### أشغل دلوقتي

```bash
python3 run_batch.py EG_ESIGN \
  --stages 1 2 3 4 5 6 7 \
  --force-reenrich
```

لأن:

* هيصلح `هيية`
* هيضيف concepts
* هيضيف applicable_to
* هيضيف explicit_cross_refs
* هيضيف target_article_id
* هيختبر الـ pipeline كاملة على قانون صغير

---

بعدها راجع الناتج النهائي بنفسك.

لو لقيت المادة شكلها بقى مثلاً:

```json
{
  "article_id": "EG_ESIGN_005",
  "concepts": [...],
  "applicable_to": [...],
  "explicit_cross_refs": [...],
  "corpus_metadata": {
      "extraction_source": "pymupdf",
      "confidence_score": 0.96
  }
}
```

ساعتها انت عملياً عندك نسخة V7 شغالة.

وبعد نجاح EG_ESIGN انتقل لـ:

```bash
EG_PDPL
```

وبعد نجاح الاثنين فقط ابدأ القانون المدني أو التجاري.

ما زلت لا أنصح إطلاقاً بتشغيل:

* القانون المدني
* التجاري
* العقوبات

قبل ما تشوف Export نهائي صحيح من EG_ESIGN و PDPL. الفرق بين إصلاح Bug على 30 مادة وإصلاحه على 1000+ مادة فرق ضخم في الوقت والتكلفة.
