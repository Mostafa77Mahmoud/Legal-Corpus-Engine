# 1) الـ cache design فيه Bug

الوكيل بنفسه اكتشف المشكلة:

> Stage 3 caches by article_id presence regardless of whether enrichment completed.

وده معناه إن الكاش كان بيقول:

```python
if article_id exists:
    skip
```

بدل:

```python
if article_is_complete:
    skip
```

وده Design Bug.

الكاش المفروض يعتمد على حالة الاكتمال، مش مجرد وجود الـ article.

أنا كنت أفضل يبقى فيه مثلاً:

```json
{
  "article_id": "...",
  "enrichment_status": "completed"
}
```

أو

```json
{
  "completed": true
}
```

أو حتى

```json
{
  "pipeline_stage": 3
}
```

بدل الاعتماد على وجود الـ JSON.

# 2) الـ Single Article Fallback محتاج تحسين

هو حصل:

```
Batch
↓

JSON Parse Error
↓

150 API calls
↓

Quota ماتت
```

وده منطقي، لكنه مش أفضل حل.

الأفضل يبقى:

```
150

↓

Parse Error

↓

Split to 75 + 75

↓

لو واحدة فشلت

↓

37 + 38

↓

...

↓

Single article
```

يعني **Binary Split Fallback**.

بدل:

```
150

↓

1

↓

1

↓

1

↓

1

↓

1
```

وده هيوفر كمية ضخمة من الـ API Calls.

---

## اطلب منه

```text
Design a hierarchical batch fallback strategy.

Current behavior:

Batch -> Single article fallback

Replace it with:

Batch
↓
Half batch
↓
Quarter batch
↓
Single article

The goal is to minimize API calls when Gemini returns malformed JSON for a large batch.

Do not implement yet.

Produce the algorithm and complexity analysis first.
```

# حاجة واحدة أنصح تضيفها دلوقتي

أنا شايف المشروع كبر.

فلازم يبقى عندك Metadata لكل مرحلة.

مثلاً:

```
pipeline_state.json
```

يبقى فيه:

```json
{
  "law_id": "EG_CIVIL_CODE",

  "stage1": "completed",

  "stage2": "completed",

  "stage3": {

      "completed_articles": 300,

      "remaining_articles":739,

      "last_article":"EG_CIVIL_CODE_337"

  },

  "stage4":"pending",

  "stage5":"pending",

  "updated_at":"..."
}
```

وده أحسن بكتير من استنتاج الحالة من الملفات.

---

## اطلب منه

```text
Design a persistent pipeline checkpoint system.

Each law should maintain a pipeline_state.json containing:

- current stage
- completed stages
- completed article count
- remaining article count
- last processed article
- total processing cost
- extraction source summary
- last update timestamp

The checkpoint must allow safe resume after:

- crash
- quota exhaustion
- manual interruption

The pipeline should never infer progress from cached files alone.

Do not implement yet.

Produce the design first.
```
