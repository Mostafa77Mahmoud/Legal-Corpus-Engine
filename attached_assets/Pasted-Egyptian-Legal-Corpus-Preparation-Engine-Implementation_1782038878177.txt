Egyptian Legal Corpus Preparation Engine
Implementation Plan — nezam-legal-corpus (Version 6)
Role: Principal Data Engineer & Legal AI Systems Architect Objective: Transform raw Egyptian law PDFs into a production-grade, validated, article-level legal corpus for the Nezam AI contract analysis platform. Version: 6.0 (Revised to include amendment-aware version registries, chunking layers, prompt versioning, structured error taxonomy, prioritized human reviews, embedding pipelines, and corpus freeze gates) Date: 2026-06-21

1. Project Goals & Scope (Revised)
Primary Goal
Build a standalone, offline data pipeline that converts raw Egyptian law PDFs into structured, enriched, and validated article objects stored in MongoDB — ready for high-fidelity routing and article-level retrieval by the Nezam AI contract analysis platform.

Scope Updates (V6)
Primary Model Choice: The entire pipeline utilizes Gemini 3.5 Flash (configured via config/settings.py -> PRIMARY_MODEL) for all LLM stages (OCR, cleanup, enrichment, semantic relationship validation, and quality auditing).
Amendment-Aware Versioning: Incorporates a separate amendment registry to track temporal states of articles, permitting historical corpus reconstructions.
Dual-Layer Retrieval Chunking: Automatically segments long articles into retrieval chunks while keeping the parent article as the canonical unit. Search queries target chunks; LLM reasoning receives the full articles.
Prompt Version Registry: Manages prompts in a dedicated schema mapping prompt versions, SHA-256 hashes, and creations. Every generated article records the prompt version.
Structured Error Taxonomy: Classifies validation errors into standardized types (e.g. ERROR_MERGED_ARTICLE, ERROR_TAXONOMY_DRIFT) for easier automated cleanup.
Prioritized Human Reviews: Implements a weighted priority score sorting flagged articles by risk severity, confidence penalty, and cross-reference failure risk.
Stage 3.7 Embedding Generation: Introduces a dedicated embedding stage generating vector representations for articles, chunks, and law metadata objects.
Corpus Freeze Release Gate: Implements a release gate blocking export until validation, benchmarking, and receipt signature conditions are satisfied.

Text Cleanup (Stage 1.3): Deterministic regex scripts normalize characters and remove watermarks, writing cleanup steps to data/cleanup_audit_logs/{law_id}_cleanup_audit.json.

6. Article Splitting & Split Validation (Stage 2.5)
Stage 2.5 executes a verification process comparing the split list against a known baseline.

python

def validate_splitting_results(law_id: str, split_articles: list[dict], expected_count: int, repealed_list: list[int]) -> dict:
    """
    Performs continuity and sequence checks on split articles.
    """
    errors = []
    warnings = []
    
    # Sort articles numerically
    sorted_arts = sorted(split_articles, key=lambda x: x['article_number'])
    
    # 1. Verify total count
    actual_count = len(sorted_arts)
    if expected_count and actual_count != expected_count:
        msg = f"Count mismatch. Expected: {expected_count}, Got: {actual_count}."
        if actual_count < expected_count:
            errors.append(msg)
        else:
            warnings.append(msg)
            
    # 2. Check for sequence gaps & duplicates
    seen_numbers = {}
    for art in sorted_arts:
        num = art['article_number']
        seen_numbers[num] = seen_numbers.get(num, 0) + 1
        
    # Check duplicates
    for num, count in seen_numbers.items():
        if count > 1:
            errors.append(f"Duplicate article number detected: {num} (appears {count} times).")
            
    # Check gaps
    max_num = max(seen_numbers.keys()) if seen_numbers else 0
    for num in range(1, max_num + 1):
        if num not in seen_numbers:
            # Check if this article was legally repealed
            if num in repealed_list:
                continue
            errors.append(f"Missing article number in sequence: {num}")
            
    # 3. Size Anomaly Check (Merged/Split Detection)
    for art in sorted_arts:
        text_len = len(art['text'].strip())
        
        # Merge Anomaly: Article too large (contains embedded article markers in body)
        if text_len > 15000:
            lines = art['text'].split('\n')[1:]
            body_text = "\n".join(lines)
            if re.search(r'(?:مادة|المادة)\s+\d+', body_text):
                errors.append(f"Potential merged article: Art {art['article_number_display']} is unusually large ({text_len} chars) and contains internal article markers.")
                
        # Split Anomaly: Article too small (incomplete or split)
        if text_len < 15:
            if "ملغاة" not in art['text'] and "مُلغاة" not in art['text']:
                errors.append(f"Potential broken split: Art {art['article_number_display']} has only {text_len} characters.")
                
    return {
        "status": "passed" if not errors else "failed",
        "errors": errors,
        "warnings": warnings
    }

Egyptian statutes are amended frequently, changing individual articles without replacing the base law. Rather than storing flat versions of the entire law, versioning lineage is stored separately in config/amendment_registry.json.

8. Retrieval-Oriented Chunking Layer (New)
While some Egyptian legal articles contain hundreds of words (e.g., listing multiple conditions or administrative paths), retrieval engines perform best with smaller, cohesive text blocks. We split articles into retrieval chunks while maintaining the parent article as the canonical document.

Chunking Rules
Automatic Generation: Generated automatically inside Stage 3.7 using a split strategy (splitting on paragraph boundaries \n\n with a maximum target size of 500 tokens and 100-token overlaps).
Linkage Retention: References and metadata are maintained at the article level; chunks inherit concepts and applicable_to parameters from the parent article to support filtering.
Execution Routing: Semantic search runs against the chunk index. The system then resolves retrieved chunk IDs back to parent article IDs, feeding the complete article text to the contract analysis LLM to prevent loss of context.

10. Structured Error Taxonomy (New)
Generic text failures make automated cleanup difficult. We replace generic validation failures with a standardized taxonomy of legal data errors.

Error Code	Definition / Detection Rule
ERROR_MISSING_ARTICLE	An expected article number in the sequential range is absent.
ERROR_DUPLICATE_ARTICLE	The same article ID appears multiple times in the database.
ERROR_MERGED_ARTICLE	Article text contains internal article markers (e.g., "مادة 148" inside Article 147 text).
ERROR_SPLIT_ARTICLE	Article body is shorter than 15 characters and contains incomplete sentences.
ERROR_CORRUPTED_TEXT	Document contains invalid control characters or high density of replacement symbols (\ufffd).
ERROR_INVALID_REFERENCE	An explicit reference points to a non-existent law or article ID.
ERROR_TAXONOMY_DRIFT	Article is tagged with concepts that do not exist in the active taxonomy.py schema.
These error keys are saved inside the validation arrays of failed articles to automate remediation steps.
